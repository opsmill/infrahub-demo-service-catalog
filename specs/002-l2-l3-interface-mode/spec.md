# Feature Specification: L2/L3 Interface Mode Selection for Dedicated Internet Service

**Feature Branch**: `002-l2-l3-interface-mode`
**Created**: 2026-03-17
**Status**: Draft
**Input**: User description: "I would like to update my dedicated internet service to allow the user to decide if they prefer to be connected to a L3 or a L2 interface, when the customer select L2, the IP address should be assigned to the vlan and the customer facing interface should be configured in L2 mode. the portal, the service and the generated config must be updated accordingly"

## Clarifications

### Session 2026-03-17

- Q: For L2 mode, should the device config render a VLAN SVI with the gateway IP on the core switch, not render the IP at all, or render an SVI on the edge router? → A: SVI on edge router. The edge router renders a VLAN interface (SVI) with the gateway IP sourced from the VLAN's associated IP addresses.
- Q: Should the transform query be extended to fetch VLAN IPs, or should the generator create an L3 SVI interface object on the router? → A: Extend the query and template. The device_info GraphQL query fetches IP addresses associated with VLANs at the device's location, and the Jinja2 template renders SVI config from VLAN data. No SVI interface object is created by the generator.
- Q: Should protocol type definition files be edited directly? → A: No. Protocol files are auto-generated by `infrahubctl protocols` and MUST NOT be edited manually. After schema changes, protocols MUST be regenerated using the appropriate command and then trimmed as needed for each target (generator protocols are trimmed to only include types needed by generators).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Customer Requests L3 Dedicated Internet (Priority: P1)

A customer accesses the service catalog portal and requests a new dedicated internet service with Layer 3 connectivity. This is the existing default behavior and must continue to work unchanged. The customer fills in their service identifier, account reference, location, bandwidth, IP package, and selects "Layer 3" as the interface mode. The system provisions the service with a gateway IP address assigned to a dedicated L3 interface on the edge router.

**Why this priority**: This is the existing behavior that must remain fully functional. Any regression here would break the current service offering.

**Independent Test**: Can be fully tested by submitting a dedicated internet request with L3 mode selected and verifying the generator creates a VLAN, prefix, L2 port on the core switch, L3 gateway interface on the edge router, and assigns the gateway IP to that L3 interface.

**Acceptance Scenarios**:

1. **Given** the service catalog portal is available, **When** a customer submits a dedicated internet request with interface mode "Layer 3", **Then** the service is created with all existing resource allocations (VLAN, prefix, L2 port, L3 gateway interface with IP) unchanged from current behavior.
2. **Given** no interface mode is explicitly selected, **When** the service is provisioned, **Then** the system defaults to Layer 3 mode to ensure backward compatibility with existing services.
3. **Given** an L3 service is provisioned, **When** the startup config is generated for the edge router, **Then** the config includes the dedicated L3 interface with the gateway IP (existing behavior, unchanged).

---

### User Story 2 - Customer Requests L2 Dedicated Internet (Priority: P1)

A customer accesses the service catalog portal and requests a new dedicated internet service with Layer 2 connectivity. The customer selects "Layer 2" as the interface mode. The system provisions the service with the gateway IP address assigned to the VLAN instead of a dedicated L3 interface. No L3 interface is created on the edge router. The customer-facing port on the core switch is configured in L2 Access mode (same as L3 mode).

**Why this priority**: This is the core new functionality requested. It enables a new service variant that customers need.

**Independent Test**: Can be fully tested by submitting a dedicated internet request with L2 mode selected and verifying the generator creates a VLAN, prefix, L2 port on the core switch, and assigns the gateway IP to the VLAN directly without creating an L3 interface on the edge router.

**Acceptance Scenarios**:

1. **Given** the service catalog portal is available, **When** a customer submits a dedicated internet request with interface mode "Layer 2", **Then** the service is created with a VLAN, IP prefix, and L2 port on the core switch.
2. **Given** a dedicated internet service is being provisioned in L2 mode, **When** the gateway IP is allocated, **Then** the IP address is linked to the VLAN (not to an L3 interface) and no L3 gateway interface is created on the edge router.
3. **Given** a dedicated internet service is being provisioned in L2 mode, **When** the customer-facing port is configured, **Then** it is set to L2 Access mode with the allocated VLAN as the untagged VLAN (same port configuration as L3 mode).

---

### User Story 3 - Portal Displays Interface Mode Selection (Priority: P1)

A customer using the service catalog portal sees a new "Interface Mode" selection field in the dedicated internet request form. The field presents two options: Layer 2 and Layer 3. The selection is clearly labeled so the customer understands the implications of their choice.

**Why this priority**: Without the portal update, customers cannot select between L2 and L3, making the backend changes unusable.

**Independent Test**: Can be tested by loading the dedicated internet form page and verifying the interface mode selector appears with both options and that the selected value is included in the service creation request.

**Acceptance Scenarios**:

1. **Given** a customer opens the dedicated internet request form, **When** the form loads, **Then** an "Interface Mode" selector is displayed with "Layer 2" and "Layer 3" options.
2. **Given** a customer fills out the dedicated internet form with L2 selected, **When** they submit the form, **Then** the service object is created with the interface mode set to "l2".

---

### User Story 4 - Startup Config Renders L2 SVI on Edge Router (Priority: P1)

When the startup configuration artifact is generated for an edge router that serves L2 mode dedicated internet services, the config includes a VLAN interface (SVI) with the gateway IP address sourced from the VLAN's associated IP addresses. This SVI is rendered from VLAN data, not from an explicit L3 interface object.

**Why this priority**: The generated device config must correctly reflect the L2 service topology. Without this, the edge router would have no gateway configuration for L2 customers.

**Independent Test**: Can be tested by provisioning an L2 dedicated internet service, then generating the startup config for the edge router and verifying it contains a VLAN SVI with the correct gateway IP and VLAN ID.

**Acceptance Scenarios**:

1. **Given** an L2 dedicated internet service has been provisioned with a VLAN and gateway IP, **When** the startup config is generated for the edge router at that location, **Then** the config includes a `interface vlan <vlan_id>` section with the gateway IP address.
2. **Given** an L3 dedicated internet service has been provisioned, **When** the startup config is generated for the edge router, **Then** the config renders the gateway IP on the dedicated L3 interface (existing behavior, unchanged).
3. **Given** a VLAN at a location has no associated IP addresses, **When** the startup config is generated, **Then** no SVI is rendered for that VLAN.

---

### Edge Cases

- What happens when an existing service (created before this feature) has no interface mode set? The system treats it as L3 mode by default.
- What happens if the interface mode value is invalid or missing during generator execution? The generator defaults to L3 behavior.
- What happens when a VLAN has multiple IP addresses associated? The SVI config renders all associated IPs.
- What happens when the same VLAN is used at a location with both L2 and L3 services? The SVI is rendered from VLAN IP data regardless; the L3 service also has its own dedicated interface.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The dedicated internet service data model MUST include an interface mode property with two options: Layer 2 and Layer 3.
- **FR-002**: The interface mode MUST default to Layer 3 to ensure backward compatibility with existing services.
- **FR-003**: The service request portal MUST display an interface mode selector in the dedicated internet request form.
- **FR-004**: When Layer 3 mode is selected, the service provisioning MUST create a dedicated L3 gateway interface on the edge router and assign the gateway IP to that interface (existing behavior).
- **FR-005**: When Layer 2 mode is selected, the service provisioning MUST assign the gateway IP address to the VLAN directly instead of to an L3 interface.
- **FR-006**: When Layer 2 mode is selected, the service provisioning MUST NOT create a dedicated L3 interface on the edge router.
- **FR-007**: In both L2 and L3 modes, the customer-facing port on the core switch MUST be configured in L2 Access mode with the allocated VLAN as the untagged VLAN.
- **FR-008**: The IP address data model MUST support a direct association with a VLAN to enable the L2 gateway IP assignment.
- **FR-009**: The service provisioning query MUST include the interface mode value so the generator can determine which provisioning path to follow.
- **FR-010**: The interface mode selection MUST be persisted as part of the service record and visible in the service details.
- **FR-011**: The device configuration query MUST fetch IP addresses associated with VLANs at the device's location.
- **FR-012**: The device configuration template MUST render a VLAN interface (SVI) with associated IP addresses for any VLAN that has IP addresses linked to it.
- **FR-013**: The device configuration template MUST continue to render dedicated L3 interfaces with their IP addresses unchanged (L3 mode backward compatibility).
- **FR-014**: Protocol type definition files MUST be regenerated (not manually edited) after schema changes using the `infrahubctl protocols` command. Generator protocols MUST then be trimmed to include only types needed by generators.

### Key Entities

- **ServiceDedicatedInternet**: Existing service entity, extended with an interface mode property (L2 or L3).
- **IP Address**: Existing entity, extended with a new optional association to a VLAN for L2 mode gateway assignment.
- **VLAN**: Existing entity, gains a reverse association from IP addresses assigned to it.
- **L3 Interface**: Existing entity, created only when service is in L3 mode. No changes to its definition.
- **L2 Interface (Port)**: Existing entity, used for customer-facing port in both modes. No changes to its definition.
- **Device Configuration (Transform)**: Existing artifact transform, extended to render VLAN SVIs with IP addresses from VLAN data.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Customers can successfully request a dedicated internet service in L2 mode through the portal and receive a fully provisioned service with the gateway IP assigned to the VLAN.
- **SC-002**: Customers can successfully request a dedicated internet service in L3 mode through the portal with no change in behavior from the current implementation.
- **SC-003**: Existing services created before this feature continue to function correctly, defaulting to L3 mode.
- **SC-004**: All schema validations pass with the updated data models.
- **SC-005**: All existing tests continue to pass without modification (backward compatibility).
- **SC-006**: The startup config generated for an edge router serving an L2 service includes a VLAN SVI with the correct gateway IP address.
- **SC-007**: The startup config generated for an edge router serving an L3 service is unchanged from current behavior.

## Assumptions

- The customer-facing port configuration (L2 Access mode on the core switch) is identical for both L2 and L3 service modes. The L2/L3 distinction only affects the gateway side (edge router).
- The VLAN, IP prefix, and port allocation logic (pool selection, random switch index) remains unchanged for both modes.
- The IP-to-VLAN relationship is a new schema extension that does not impact existing IP address records (the relationship is optional).
- The device config transform renders SVIs for all VLANs that have associated IP addresses, regardless of which service mode created them. This is a data-driven approach: if a VLAN has IPs, it gets an SVI.
- Protocol files (`protocols_sync.py`, `protocols_async.py`, `generators/protocols.py`) are auto-generated and MUST NOT be edited manually. They are regenerated from schemas using `infrahubctl protocols` and trimmed as needed.
