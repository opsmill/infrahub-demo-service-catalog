Data Model
Schema Domains Loaded into Infrahub
The project uses a modular, layered schema covering network infrastructure, IP address management, locations, VLANs, and service definitions.

DCIM

  • Device, DeviceType, Platform
  • InterfaceL2, InterfaceL3
  • Endpoint, Connector
  • Roles: core, edge, spine, leaf, cpe

IPAM

  • IPAddress (with FQDN)
  • Prefix (with status, role, gateway)
  • Inherits from BuiltinIPAddress/IPPrefix
  • Resource Pool support

VLAN

  • VLAN (ID from Number Pool)
  • L2 Domain
  • Relationships to prefixes & interfaces
  • Location-aware VLAN assignment

Location

  • Country → Metro → Site → Rack
  • Hierarchical parent/children
  • LocationHosting for devices & VLANs
  • Facility IDs, physical addresses

Organization

  • Manufacturer (linked to DeviceType)
  • Provider (circuits & locations)
  • Generic base with tags

Service

  • ServiceGeneric base (identifier, account)
  • DedicatedInternet (bandwidth, IP package)
  • Status lifecycle: draft → active
  • Links to interface, VLAN, prefix, gateway
# Example: Service Schema (service.yml)
nodes:
  - name: DedicatedInternet
    namespace: Service
    inherit_from: [ServiceGeneric]
    attributes:
      - name: status       # draft → in-delivery → active → decommissioned
        kind: Dropdown
      - name: bandwidth    # 100 Mbps, 1 Gbps, 10 Gbps
        kind: Dropdown
      - name: ip_package   # small (6 IPs), medium (14), large (30)
        kind: Dropdown
      - name: interface_mode # Layer 2 or Layer 3
        kind: Dropdown
    relationships:
      - name: location             # → LocationSite
      - name: dedicated_interfaces  # → DcimInterface (many)
      - name: vlan                 # → IpamVLAN
      - name: prefix               # → IpamPrefix
      - name: gateway_ip_address   # → IpamIPAddress
Workflow
Service Delivery Lifecycle
From a user clicking "Request Service" to configuration deployed on network devices — this is the end-to-end flow powered by Infrahub generators and branch-based change management.

Service Request via Portal

A user selects a site, bandwidth, IP package, and interface mode through the Streamlit self-service portal. The request is submitted to Infrahub via its Python SDK.

Streamlit + Infrahub SDK

Service Object Created in a Branch

A ServiceDedicatedInternet object is created in an Infrahub branch with status draft. This isolates changes from production data until they are reviewed and merged.

Infrahub Branch

Generator Executes Automatically

The DedicatedInternetGenerator runs server-side, performing automated resource allocation: a VLAN from the number pool, an IP prefix from the prefix pool, a free physical port on the site's core switch, and a gateway IP on the edge router.

Infrahub Generator

Proposed Change & Review

A Proposed Change (similar to a pull request) is created in Infrahub, letting the team review all allocated resources, data integrity checks, and rendered configs before merging to main.

Infrahub Proposed Change

Configuration Rendering

Jinja2 templates combined with GraphQL queries render device-specific configs (VLANs, SVIs, interface settings) as Infrahub artifacts, ready to be consumed downstream.

Jinja2 + GraphQL

Deployment via Ansible & Semaphore

Ansible playbooks fetch the rendered configuration artifacts from Infrahub using the opsmill.infrahub collection, then push configs to devices. Semaphore provides a web UI to trigger and monitor deployment runs.

Ansible + Semaphore
Infrahub Features Used
Key Platform Capabilities
This project exercises many of Infrahub's core features, demonstrating the platform's value as a unified infrastructure management system.

Extensible Schema System

YAML-based schemas with generics, inheritance, extensions, and relationships. Models are versioned alongside code in Git.

Branch-Based Change Management

Git-like branching for infrastructure data. Proposed Changes provide review and validation before merging — no accidental changes in production.

Generators

Server-side Python scripts that react to data changes and automate complex multi-step provisioning workflows like resource allocation.

GraphQL API

Full-featured GraphQL API for querying and mutating data. Used by the frontend, generators, and Jinja2 templates alike.

Resource Pools

Number pools (VLANs) and IP prefix pools provide deterministic, conflict-free allocation of network resources across concurrent requests.

Artifact Rendering

Jinja2 templates + GraphQL queries produce device configurations stored as versioned artifacts, bridging the SoT with network automation tools.

Git Integration

The repository is integrated as a CoreRepository in Infrahub. Schema changes, generators, templates, and transforms are all version-controlled.

Python SDK

The infrahub-sdk provides a typed, async Python API for creating, querying, and managing objects — used in both the portal and the generators.
